## 项目模块功能扩展规划

### 用户与权限服务

#### 第一月版本问题分析

第一月版本中，用户与权限服务存在以下核心痛点：

1. **登录体验单一且繁琐**：仅支持平台自有账号密码登录，未集成主流社交账号或科研平台账号（如 GitHub、Google Scholar）登录能力，导致用户需重复注册账号，尤其对于习惯跨平台使用的科研人员，操作效率低且易产生账号记忆负担。
2. **用户科研属性刻画不足**：个人资料仅包含基础信息（如姓名、单位），缺乏科研领域标签、学术成果关联等专业属性，导致平台无法精准匹配用户与项目（如 “机器学习” 方向的用户难以被推荐至相关项目），团队协作中成员背景认知成本高。
3. **组织关系管理缺失**：未支持多组织层级与用户多身份关联，无法适配高校 “学校 - 学院 - 实验室” 的多层级架构，跨院系协作时用户身份归属混乱，项目权限继承规则不清晰（如实验室成员无法默认获取实验室公开项目的访问权限），管理员需手动维护权限，效率低下。



#### 模块需要扩展的功能和解决的需求

- **用户注册/登录**
  - **新增需求：**
    - 解决用户已有其他的身份认证，重复注册登录繁琐的痛点，提升登录便捷性，同时支持主流社交账号快速登录。
  - **对应的功能：**
    - **第三方账号登录**：通过 OAuth2.0 协议集成第三方平台的登录实现，登录后可关联已有平台账号或自动创建新账号。
    - **账号绑定管理**：用户可在 “个人设置 - 账号安全” 中查看已绑定的第三方账号，支持解绑或新增绑定，避免账号孤立。
- **个人资料精细化管理**
  - **新增需求：**
    - 解决 “用户科研属性不明确，项目匹配和团队协作效率低” 的问题，丰富用户画像，辅助科研协作。
  - **对应的功能：**
    - **研究方向标签体系**：
      - 支持用户添加 1-5 个研究方向标签（如 “机器学习”“量子计算”），标签可从系统预设标签库选择或自定义（需审核，对接平台管理模块）。
      - 标签支持层级结构（如 “人工智能”→“自然语言处理”→“大模型”），便于精准分类。
    - **学术成果关联**：用户可手动关联所在项目内**公开**的成果，例如已发表论文、专利等学术成果（关联 “数据与模型仓库服务” 中的成果 ID），展示个人学术背景。
    - **个性化设置**：可以查看其他成员的个人主页，支持自定义个人主页可见范围（如仅团队可见、仅项目成员可见、公开），控制资料隐私。

- **组织与院系管理**
  - **新增需求：**
    - 为了解决用户的多重组织的身份关系，解决 “高校多院系、多实验室层级管理混乱，跨组织协作效率低” 的问题，适配高校行政架构，需要对用户的组织关系进行维护。
  - **对应的功能：**
    - **组织架构维护**：支持用户创建（需要审核，对接平台管理模块）或者加入已有的组织（需要组织的创建者同意），每个组织有唯一的标识和负责人（可以多名）。
    - **用户组织关联**：用户可被分配至多个组织，关联后继承组织所在项目的默认权限（如查看组织内公开项目）。组织负责人自动拥有该组织下所有项目的权限，便于宏观管理。



#### 扩展功能价值总结

1. **提升用户登录便捷性与平台粘性**：通过 OAuth2.0 集成第三方登录，减少注册门槛，预计降低新用户流失率 30%；账号绑定功能避免用户因多账号孤立导致的信息分散，提升用户留存。
2. **强化科研协作精准度**：研究方向标签体系与学术成果关联丰富用户科研画像，平台可基于标签实现项目 - 用户智能匹配，提升团队组建效率；个人主页权限控制满足科研数据隐私需求（如未公开成果仅团队可见），平衡开放性与安全性。
3. **适配高校组织架构，降低管理成本**：组织架构维护与多身份关联功能贴合高校行政体系，实现 “用户 - 组织 - 项目” 权限的自动继承（如实验室成员默认获取实验室项目访问权），减少管理员手动配置工作量，跨组织协作效率提升 40%。



### 项目与团队管理服务

#### 第一月版本问题分析

第一月版本中，项目与团队管理服务存在以下核心痛点：

1. **项目进度管控低效**：缺乏可视化仪表盘，项目负责人需手动汇总任务数据（如完成率、延期情况），无法实时掌握项目全局状态；关键节点（里程碑）无时间线展示，易遗漏重要阶段目标。
2. **操作溯源与合规性不足**：未记录用户在项目内的关键操作（如任务修改、成果删除），当发生数据误操作或安全事件时，无法追溯责任主体，不符合高校科研数据 “可审计、可追溯” 的合规要求。
3. **团队协作存在信息孤岛**：项目内讨论依赖外部工具（如微信、邮件），重要沟通内容未沉淀至平台，新成员加入后难以快速了解项目背景；成员贡献度缺乏量化指标，无法客观评估团队协作效率。
4. **权限与任务管理粗糙**：项目权限集中于创建者，多负责人场景下（如联合导师）无法共同管理项目，权限分配灵活性不足；任务完成仅通过状态修改确认，缺乏提交详情与审核流程，任务质量难以把控，易出现 “虚假完成” 情况。



#### 模块需要扩展的功能和解决的需求

- **项目进度可视化与数据分析模块**
  - **新增需求**
    - 满足导师 / 负责人对项目宏观进度的把控需求，提供量化指标支撑决策。优化项目进度的可视化，为项目进度提供更良好的可视化仪表盘展示，提供各种关键指标统计，指出任务完成率、成员贡献度等
  - **对应的功能：**
    - **项目仪表盘（Dashboard）**：
      - 顶部指标区：展示项目总任务数、已完成数（完成率）、延期任务数、本周新增任务数。鼠标悬停图表元素时，显示详细数据
      - 图表区：任务状态分布饼图（TODO/IN_PROGRESS/BLOCKED/DONE）、成员任务负载柱状图（按分配任务数）、进度趋势线（近 30 天每日完成任务数）。鼠标悬停图表元素时，显示详细数据
      - 里程碑时间线：可视化展示项目关键节点（如 “方案评审”“数据采集完成”），支持手动添加 / 编辑，逾期节点标红提醒。鼠标悬停图表元素时，显示详细数据，点击里程碑可查看关联任务列表。
      - 成果统计：拥有对成果统计更加详细的面板，自动同步本模块数据，显示 “项目内成果总数、已发布数、本月新增数等信息”。
- **用户操作日志**
  - **新增需求：**
    - 为了解决用户 “操作溯源困难，安全事件无法追踪，项目内成员的操作没有详细的记录” 等问题，满足高校科研数据安全合规要求。

  - **对应的功能**：
    - **关键操作日志记录**：自动记录用户登录（含 IP、地区等详细信息）、对项目的相关操作、任务相关操作、项目内成果操作、项目内wiki文档操作等行为，包含操作人、时间、详情、结果等详尽的信息
    - **日志查询与导出**：项目内成员可以查看操作日志，可以查询和按条件筛选操作日志，支持导出为 Excel。项目管理员（所有者）额外添加可以对操作日志进行操作的功能。

- **团队协作与讨论区**
  - **新增需求：**
    - 为了方便量化成员贡献，提升项目内沟通效率，减少信息孤岛。

  - **对应的功能**：
    - **项目内讨论区**：项目内新增专供于项目成员的讨论区，把重要的讨论内容持久化可视化，留给项目全体成员方便可见的重要内容的讨论区域，讨论内容支持 Markdown 格式，支持上传附件。项目负责人可以对项目讨论区进行管理。

- **项目权限内容修改**
  - **新增需求：**
    - 为了解决项目内只有项目创建者有最高权限的问题，这不利于多负责人存在的情况下对项目的统筹管理。

  - **对应的功能：**
    - 修改项目创建人为项目管理员，拥有权限不变，项目管理员可以指定项目内的成员为管理员，并且对权限进行如下补充，项目管理员可以邀请成员和踢出成员。

- **任务完成调度和审核**
  - **新增需求**
    - 需要解决项目内项目的完成太过于粗糙，不够详尽，可视化程度低，所以项目创建的时候需要编辑信任务的同时，项目完成也需要提交详细的信息
    - 任务管理过于依赖线下交流，没有完整的线上管控管理

  - **对应的功能：**
    - 添加任务提交详情内容，用户在修改任务状态为完成的时候，需要提交相应的信息，支持图片和附件，实现详细的提交完成任务的内容
    - 添加任务的审核和待审核功能，用户在完成任务并且提交信息之后，创建任务的用户会收到审核信息，此时的任务状态为待审核，创建任务的用户需要同意通过审核才能最终修改为已完成




#### 扩展功能价值总结

1. **为管理者提供数据驱动的决策支撑**：项目仪表盘通过量化指标（任务完成率、成员负载）与可视化图表，帮助导师 / 负责人实时把控项目进度，逾期节点标红提醒功能可降低关键任务延误风险，决策效率提升 50%。
2. **强化团队协作效率与知识沉淀**：项目内讨论区实现沟通内容结构化沉淀，Markdown 与附件支持满足科研场景下公式、图表的表达需求；成员贡献度分析为绩效考核提供客观依据，减少协作摩擦。
3. **提升权限灵活性与任务质量**：多管理员权限设置适配联合管理场景，任务提交详情与审核流程确保任务完成质量，避免 “形式化完成”，任务返工率降低 35%。
4. **满足科研合规要求**：操作日志记录覆盖全流程关键行为，支持查询与导出，符合《科研数据管理规范》中 “可追溯” 要求，降低平台合规风险。



### 数据与模型仓库服务

#### 第一月版本问题分析

结合第一版本核心功能与科研团队实际使用场景，数据与模型仓库服务存在以下需优化的核心问题：

1. **AI 赋能助手专业性与可信度不足**：AI 辅助功能未区分场景（如论文撰写、研究分析），生成内容缺乏学术来源引用，科研人员对 AI 建议的采信度低；无法基于用户研究领域调整回答专业度，通用性回答难以满足细分领域需求（如 “量子计算” 与 “生物信息” 的专业术语差异）。
2. **Wiki 协作效率低**：仅支持单人编辑 Wiki 文档，多人同时编辑时易产生内容冲突（如覆盖他人修改），离线协作需频繁手动合并版本，导致知识库共建效率低下。
3. **文件管理体验差**：大文件（如实验数据集、模型权重）上传易因网络中断失败，需重新上传；缺乏批量操作功能，对多文件成果（如论文 + 补充材料 + 代码）的管理耗时；支持预览的文件类型有限（仅基础文档），科研专属文件（如.ipynb、.mat）需下载后查看，影响效率。
4. **数据链条断裂**：成果（如论文、专利）与任务、项目的关联关系未打通，无法追溯 “成果源于哪个任务”“任务属于哪个项目”，知识沉淀与项目执行形成信息孤岛，不利于后续研究复用。



#### 模块需要扩展的功能和解决的需求

- **AI赋能助手调整**：
  - **新增需求：**
    - 提升 AI 辅助的专业性与可信度，让科研人员更愿意依赖 AI 工具提升效率。

  - **对应的功能：**
    - 支持用户选择 AI 赋能模式，支持论文模式，在论文模式上能够更加准确的给出论文的来源，能够生成更加准确专业的摘要，能够支持给出可以直接跳转的论文来源链接
    - 支持用户选择研究赋能模式，用户给出自己的研究领域，AI模型能够根据研究领域的不同进行调整，给出在其对应领域的辅助研究，回答如果使用引用，需要显示引用的信息（如 “引用：已发布论文《XXX》”“引用：草稿报告《XXX》（仅供参考）”）。

- **wiki协同编辑**：
  - 新增需求：
    - 支持Wiki文档的编辑成本，解决离线多人合作的不变并且降低多人协作时的冲突成本，支持在线协同的编辑，让团队实时同步文档思路，提升知识库共建效率。

  - 对应的功能：
    - **支持Wiki文档多人协作**：支持项目内成员能够对Wiki文档进行多人在线的协同编辑，并且力求较好的编辑体验，每个人编辑完成后能够随时提交。
    - **实时协作感知**：编辑页面显示当前在线编辑者的头像及编辑位置，鼠标悬停某段落时提示 “XX 正在编辑此处”。

- **成果文件管理优化：**
  - 新增需求：
    - 提升大文件 / 批量文件的管理效率，增加用户在管理文件的方便程度

  - 对应的功能
    - 大文件上传支持断点续传：网络中断后重新连接，自动从断点继续上传，显示实时进度（百分比 + 剩余时间）。
    - 支持批量上传 / 下载：用户可以对一个成果内进行文件的批量上传，可以在一个成果内批量下载成果内的文件，而且可以支持批量下载多个成果（成果内的全部文件）
    - 支持常见文件类型的预览：支持对文本文件的在线预览，txt，markdown等，支持对图片的在线预览，支持对excel的在线预览，支持对常用科研文件预览（如.ipynb 代码文件、.mat 数据文件、.pdf 文件），无需下载即可查看内容。

- **跨模块的数据联动**
  - 新增需求：
    - 打通 “成果 - 任务 - 项目” 的数据链条，让知识沉淀与项目执行形成闭环，提升信息追溯效率。

  - 对应的功能
    - **成果与任务关联**：创建 / 编辑成果时，支持选择 “关联任务”（从当前项目的任务列表中选择），关联后任务详情页显示 “关联成果”，成果详情页显示 “来源任务”。例如，“论文《XXX》关联至任务‘完成核心实验流程的论文撰写’”




#### 扩展功能价值总结

1. **提升 AI 助手的科研实用性**：场景化 AI 模式（论文模式、研究模式）结合来源引用，增强建议可信度，预计科研人员对 AI 工具的使用率提升 40%；领域化调整功能确保回答贴合细分研究方向，减少专业术语误差。
2. **优化 Wiki 协作体验**：多人在线协同编辑与实时感知功能（编辑者位置提示）避免版本冲突，编辑效率提升 60%，加速团队知识库沉淀。
3. **提升文件管理效率与便捷性**：断点续传降低大文件上传失败率；批量操作与多类型预览（含科研专属文件）减少用户操作步骤，文件管理时间缩短 50%。
4. **构建 “项目 - 任务 - 成果” 数据闭环**：关联功能实现全流程追溯，方便科研人员快速定位 “成果对应的任务背景”，促进研究成果复用与传承，知识沉淀效率提升 35%。



### AI 实验分析助手

#### 第一月版本问题分析

第一月版本中，AI 实验分析助手存在以下核心痛点：

1. **RAG 功能体验受限**：每次对话需重新上传文件，无法在长回合对话中保持文件上下文，对于 “基于同一批实验数据进行多轮分析” 的场景（如 “先解读数据趋势，再预测后续结果”），用户需重复上传，操作繁琐且消耗服务资源。
2. **实验数据解析能力缺失**：无法自动解析科研数据文件（如.csv、.mat），用户需手动提取数据特征后输入 AI，导致数据分析流程割裂，无法实现 “上传即分析” 的一站式体验。
3. **实验设计与报告支持不足**：实验设计依赖用户经验，缺乏系统性优化建议（如样本量合理性、对照组设置），易出现设计缺陷；实验报告需手动撰写，格式繁琐且专业表述不统一，消耗科研人员大量时间。



#### 模块需要扩展的功能和解决的需求

- **RAG功能的深度扩展：**
  - **新增需求：**
    - 基础的基于用户的上传本地文件和基于知识库内容的智能问答有一定的局限性，AI助手不能深度，长回合对话的读取其中的文件并进行RAG，每次对话都要重新上传文件，对服务和用户体验都不够友好
  - **对应的功能**
    - AI 实验分析助手能够在与用户的一轮对话内，一直使用用户已上传的文件列表内的文件进行RAG，而且用户随时可以调整（添加文件，删除文件）已上传的文件列表，下回合对话的AI RAG的回复要根据已上传的文件列表进行调整
- **AI能解析文件数据分析：**
  - **新增需求**
    - 解决"实验数据分散、AI难以使用分析工具分析，需要用户完全的自己分析"的痛点，需要提供一站式的实验数据分析平台，满足科研人员对实验结果的深度挖掘需求
  - **对应的功能：**

## 项目新增模块规划

### 我的活动

#### 模块需要解决的需求

**核心痛点**

1. **信息分散**：用户需要频繁切换不同项目查看任务和动态
2. **审核效率低**：审核任务分散，容易遗漏或延迟处理
3. **缺乏全局视图**：无法快速了解个人在所有项目中的工作负载
4. **优先级管理困难**：跨项目任务缺乏统一的优先级排序

**核心需求**

- **信息聚合**：将用户在多个项目中的任务、审核、动态集中展示，减少项目间切换，集中处理待办事项
- **统一操作入口**：提供集中的任务处理和工作流操作界面
- **优先级管理**：支持跨项目任务的优先级排序和筛选
- **工作负载可视化**：直观展示个人工作量和时间分布



#### 模块功能定位

本模块是用户的**个人工作指挥中心**。它聚合了用户在平台所有项目中的关键工作内容，提供统一的操作界面和智能排序，帮助用户高效处理跨项目事务。其核心目标是**减少上下文切换，提升个人工作效率，确保重要事项及时处理**。

**用户故事：**

- **作为一名参与多个项目的研究生**，我希望在一个页面看到所有需要我处理的任务和审核，按优先级排序，避免遗漏。
- **作为一名团队负责人**，我希望快速查看我管理的所有项目的待审核任务，批量处理提高效率。
- **作为一名忙碌的科研人员**，我希望有一个仪表盘展示我本周的工作重点和截止日期，合理安排时间。



#### 模块拥有的功能

- **集中任务审核部分**：
  - **聚合审核任务**：
    - 聚合展示用户在所有项目中需要审核和待审核的任务。把任务审核在其中集中实现和展示，集中管控审核和待审核的任务，方便多项目成员统一管理任务，提升协作效率。
    - 支持按项目、优先级、截止日期筛选排序，逾期任务红色高亮显示
- **个人工作仪表盘**
  - **项目总览：**
    - 聚合用户参与的所有项目，点击查看项目的详情信息，可以选择跳转到项目广场进行详细的处理
  - **任务总览**：
    - 展示待处理、进行中、已完成任务数量，能够按项目分布展示任务并且进行任务统计，按能够时间顺序展示近期任务安排
    - 展示用户一周内任务完成的情况统计
    - 支持调整为统一任务列表，能够聚合所有项目中的个人任务，支持按状态，截止日期、优先级筛选排序
- **个人操作日志展示**
  - 个人动态展示：
    - 展示个人在各个项目的操作记录，能够记录任务完成、成果提交、评论互动等用户操作日志（对应上述用户操作日志，只不过只展示自己的内容）

#### 模块价值总结

1. **提升个人工作效率**：信息聚合功能减少跨项目切换频率，用户处理待办事项的时间缩短 30%；优先级排序与逾期高亮确保重要任务（如审核请求）不被遗漏。
2. **优化多项目管理体验**：个人工作仪表盘直观展示跨项目任务分布与完成情况，帮助用户合理分配时间（如避免某项目任务堆积），工作负载均衡度提升 40%。
3. **强化个人操作追溯**：个人动态展示功能方便用户回顾历史操作（如 “上周提交的成果是否已审核”），减少记忆负担，操作溯源效率提升 50%。



### 消息与通知服务

#### 模块需要解决的需求

第一月版本中，缺乏统一的消息与通知机制，导致以下痛点：

1. **关键信息触达低效**：任务分配、审核请求等重要事件通过项目内零散提示告知，用户易因未查看项目而遗漏（如 “被分配任务后 3 天未处理”），影响协作流程推进。
2. **通知类型混乱**：不同业务场景的通知（如系统安全提醒、项目状态变更）混在一起，用户难以快速识别高优先级信息（如 “任务逾期警告” 与 “新功能上线通知” 无区分）。
3. **通知管理缺失**：无法筛选、删除或标记已读通知，历史通知堆积导致重要信息被淹没；通知保存周期不明确，用户需手动记录关键信息以防丢失。



#### 模块功能定位

消息与通知服务是连接平台各业务模块（用户权限、项目管理、数据仓库等）与用户的**核心枢纽**，负责将关键业务事件（如任务分配、评审请求、成员申请等）以精准、及时、可管理的方式触达用户。

**核心价值**：解决第一版本中 “通知分散、触达低效、用户易遗漏关键信息” 的痛点，通过统一的消息中心和智能提醒机制，降低团队沟通成本，确保科研协作流程顺畅推进。



#### 模块拥有的功能

- **关键操作发送通知：**

  - 拥有覆盖全平台业务场景的通知类型体系

    基于现有模块的业务事件，梳理通知类型及触发场景：

    | 通知大类     | 具体类型         | 触发场景                                                     | 优先级 |
    | ------------ | ---------------- | ------------------------------------------------------------ | ------ |
    | **任务相关** | 任务分配         | 负责人创建任务并指派给成员，通知该成员                       | 高     |
    |              | 任务状态更新     | 任务执行者修改状态（如 “进行中→待审核”）                     | 中     |
    |              | 任务审核结果     | 审核人通过 / 驳回任务完成申请                                | 高     |
    |              | 任务截止提醒     | 任务截止前 1 天 / 3 天（可配置）                             | 中     |
    |              | 任务逾期警告     | 任务超过截止日期未完成                                       | 高     |
    | **项目相关** | 成员加入申请     | 学生申请加入公开项目，通知项目负责人                         | 中     |
    |              | 成员加入审批结果 | 负责人同意 / 拒绝申请，通知申请者                            | 中     |
    |              | 角色权限变更     | 负责人修改成员在项目中的角色（如 “参与者→核心成员”），通知所有成员 | 中     |
    |              | 项目状态变更     | 例如，项目从 “进行中→已归档”，通知所有成员                   | 低     |
    | **成果相关** | 成果评审请求     | 创作者提交成果评审，通知评审人                               | 高     |
    |              | 成果状态变更     | 成果状态发生变动，通知成果的上传者                           | 中     |
    |              | 成果发布通知     | 成果最终发布，通知项目所有成员                               | 低     |
    | **系统相关** | 账号安全提醒     | 异地登录、密码修改、第三方账号绑定变更等                     | 高     |
    |              | 广播信息         | 平台管理员可自定义广播信息，例如新功能上线（如 “AI 助手升级”），通知平台所有用户 | 低     |

  - **使用站内信渠道通知分发**

    - 支持用户根据场景选择通知渠道，确保关键信息触达，通过**站内信**作为平台内消息中心的核心通知方式
    - 在**消息列表页**进行消息展示，点击显示完整内容，支持筛选，搜索，批量删除和清除提醒（全部已读），支持优先级，所有类型通知默认同步，默认保存180天内的信息。用户可以手动清理。

- **智能提醒机制**：

  - **分级提醒**：高优先级通知（如审核请求）触发不同程度的用户提示
  - **未读提示**：支持提示用户未阅读的消息，带有特殊样式或者提示



#### 模块价值总结

1. **确保关键信息精准触达**：覆盖全业务场景的通知体系结合优先级分级，高优先级事件（如审核请求、任务逾期）触达率提升至 95%，避免因信息遗漏导致的协作延误。
2. **提升通知管理效率**：消息列表支持筛选、搜索与批量操作，用户查找特定通知的时间缩短 60%；180 天自动保存与手动清理功能平衡信息留存与存储空间。
3. **降低团队沟通成本**：统一消息中心替代分散的提示渠道，减少用户 “跨模块查通知” 的操作，团队协作响应速度提升 40%，尤其适合多项目并行的科研场景。
